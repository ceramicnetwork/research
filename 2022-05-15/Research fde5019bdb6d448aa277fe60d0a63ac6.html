Research

Welcome to our home for technology research at 3Box Labs. This page aims to provide a living overview of Ceramic’s systems design, research goals, and open problems. There is a detailed description of the architecture and its components, current and future implementations of the protocol, and a list of open problems. All research is organized into a four stage process.

Goals

At a high level the goal of our research is to understand what it would take to make Ceramic a truly web scale for application data. Blockchains today are quite far of the mark, mainly because they focus on financial security. However, not even storage focused blockchains reach the scale needed (see: 🏚️Why existing solutions are insufficient). We need a vastly improved approach to decentralized data systems.

Scalability

We define scalability as the data throughput of the network, defined as byte per second. There is of course a lot of nuance here. For example, are we talking about pure data throughput or also including some sort of processing on top of the data? Our goal is to identify scaling bottlenecks across the stack and address them.

Resilience

The resilience of the network can be thought of in different ways. How secure is the network against various sort of attackers? How decentralized is the network? Can the network tolerate partitions?

Economics

What does it mean for the economy of the network to be sustainable? Can incentives cover the cost of miners? Can the cost for users remain low? How does the FIRE token capture value?

Process overview

Research is organized into four different stages as described in the figure 1. Components / implementations represent the current state of things. Open problems point out flaws or areas of further research. Proposals are suggestions of how to address problems. Finally decisions simply state if particular proposals should be adopted or rejected.

Stages

Each stage has a separate database. Below you can find a description of each stage and relevant links to get started.

🔸Components

The components DB aspires to be a detailed specification of the entire Ceramic stack. From a top level architecture overview to detailed descriptions of minor components. The information contained within this DB should equip an implementor with everything they need in order to successfully implement one or multiple layers of Ceramic.

🛠️Implementations

The implementations DB is meant as a lightweight overview for all implementations of the three Ceramic layers. Each entry represents a code base and has a reference to each component it implements.

❓Open Problems

The open problems DB aims to highlight all open problems that currently exists within the 🔸Components specs, as well as problems related to specific 🛠️Implementations. Each item in the DB should contain enough information to understand the problem, and may be used as a living document for further research.

💡Proposals

Proposals should contain well specified changes to one or more 🔸Components and/or 🛠️Implementations. For changes that touch changes to components a complete list of changes component by component is required. For changes that touch implementations draft project specs should be included.

🔨Decisions

Decisions are created to define a binary outcome in regards to the adoption of one or more 💡Proposals. A decision can be moved to accepted status when it has been accepted in a meeting of all affected core-devs.

Miscellaneous

📸Research snapshots

The snapshots DB contains links full HTML shapshots of the 🧫Research page, hosted on github. Effectively enabling versioned history of all of our research.

In order to make a new snapshot:

  1. Use Notion’s export functionality to export as HTML, make sure to check “Include subpages”
  1. Rename the downloaded folder to todays ISO formatted date (i.e. YYYY-MM-DD)
  1. Upload the folder to the ceramicnetwork/research github repo
  1. Create a new entry in the 📸Research snapshots DB including the url to the uploaded page
